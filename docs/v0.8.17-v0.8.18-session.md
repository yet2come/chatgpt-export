# v0.8.17 / v0.8.18 セッションまとめ (2026-05-12 → 13)

このドキュメントは PR #15 のスコープ拡張に至った 1 セッション分の作業を時系列でまとめた **postmortem** です。詳細は逐次該当ファイルへリンクします。

## 0. 出発点

- 前段の PR #15 (commit `d5721d0`) では `_bulk-run-stats.jsonl` から 2 点を追加して N=7 → N=9 に拡張し、飽和モデル `C_max=275, τ=115` を得て線形仮説を棄却した。
- 残課題:
  - **中 gap (40-90 分)** が観察データに薄い
  - **時刻軸 H2** は弱く棄却したが日中サンプルが足りない
  - asset 取得失敗 (`HTTP 404` / `no_signed_url`) の理由が `_bulk-asset-failed.log` から特定できない
- この状態で v0.8.17 セッションを開始した。

## 1. v0.8.17 — asset 取得失敗 diag 列の追加

### 動機

`_bulk-asset-failed.log` は TSV 6 列 (`timestamp / convId / assetBase / source / composite / reason`) で reason だけ残しており、

- `no_signed_url` が「削除済」「期限切れ」「権限切れ」のどれなのか分離不能
- `HTTP 404` が sediment の TTL 切れか別原因か区別不能

という診断不足だった。

### 実装

- TSV に 7 列目 `diag` を追加。`HTTP <status>` / `no_signed_url` / `HTML_response` 発生時に backend response body の先頭 500 文字 (tab/改行除去) を残す。
- `downloadAssetBlob` の戻り値を `{ blob, reason }` → `{ blob, reason, diag }` に拡張。
- `r.json()` を `r.text()` + `JSON.parse` に置換 (body を diag に残しつつ parse するため。`.json()` は body 1 回しか読めない)。
- `recordAssetSkip(reasonTag, diagTag='')` で diag を受け取れるよう拡張。`HTML_response` 時は blob 先頭 512 バイトをデコードして 200 文字まで diag に記録。
- 後方互換: 既存 TSV を `awk -F$'\t'` で 1〜6 列読むスクリプトは無修正で動く (空 diag は末尾タブのみ)。

該当ファイル: [chat-bulk-export.js](../chat-bulk-export.js) (`captureBodyText`, `downloadAssetBlob`, `recordAssetSkip`)

## 2. 3 ラン実行と N=12 拡張

v0.8.17 を同じアカウントで連続 3 セッション実行し、`_bulk-run-stats.jsonl` に 3 点を追加した:

| run | UTC 開始 | UTC 終了 | JST 開始 | 直前 gap | `preFirstThrottleSuccessCount` | 終了理由 |
|---|---|---|---|---:|---:|---|
| Run A | 2026-05-12 02:56Z | 04:22Z | 11:56 | 183 分 | 246 | maxBatchPauseMs |
| Run B | 2026-05-12 11:39Z | 12:38Z | 20:39 | 437 分 | 239 | maxBatchPauseMs |
| Run C | 2026-05-12 13:58Z | 14:26Z | 22:58 |  80 分 |  95 | 完走 (pause なし) |

3 ラン合計 272 + 257 + 108 = 637 件 (1 件は `update_time` 変動による再エクスポート) で **全 636 件 done**。

これを Run 1/2 と合わせて飽和モデルを N=9 → N=12 に再フィット:

| モデル | パラメータ (N=9) | R² (N=9) | パラメータ (N=12) | R² (N=12) |
|---|---|---:|---|---:|
| 飽和 | `275 × (1 - exp(-gap/115))` | 0.958 | `271 × (1 - exp(-gap/116))` | 0.933 |

`C_max` / `τ` は実質不変、R² は微低下。詳細表とプロットは [docs/observation-experiment-1.md §回帰](observation-experiment-1.md) を参照。

## 3. Run C のアンダーシュート (40 件) — 3 仮説併記

Run C (gap 80 分 / 95 件) が新飽和モデル予測 135 件から **−40 件のアンダーシュート**。N=12 fit の R² 微低下はほぼこの点に由来する。

### 仮説 1: pause penalty

Run B が `maxBatchPauseMs` で強制 pause した直後に Run C を始めた。pause で終わるということは server が「これ以上は与えない」と判断した状態で、そこからの bucket 回復は単純な指数より遅い可能性。

### 仮説 2: workload composition

Run C が処理した 108 件は **更新順末尾 = 最古会話**で、asset (画像 / 添付) がほぼ無い。1 会話あたり `/backend-api/conversation/<id>` 1 回で完結する。一方 Run 1/A/B は新しい会話で各会話 `/files/<id>/download` を多数叩いている。

server bucket が `/backend-api/*` 全体共有モデルだと、API 呼び出し総数で見ると Run 1/A/B の方が圧倒的に消費している。すなわち Run C の「95 件で throttle」は「真の bucket は同等だが per-conv の API 呼び出しが少ないので `/conversation/` 単位の throttle が早く起きた」可能性。

### 仮説 3: 混合

両方の効果が重なっている。

### 切り分け案

- 仮説 1 検証: **自然完走後の short gap** で次 run の preFirstThrottle を観測
- 仮説 2 検証: **最古 N 件だけを scope** にして fresh run、または manifest の `stats.saved` / `stats.jsonAssetCount` を変数化して多変量回帰

現データでは切り分け不能。詳細は [docs/observation-experiment-1.md §Run C アンダーシュートの解釈](observation-experiment-1.md) を参照。

## 4. diag 列の集計結果 (147 件)

3 ラン横断の asset 失敗 147 件の diag 分布:

| reason | diag (error_code) | 件数 | 解釈 |
|---|---|---:|---|
| `no_signed_url` | `file_not_found` | 116 | user upload (`file-…`) の実体消失 |
| `no_signed_url` | `permission_error` | 1 | 共有取り消し / 古い composite asset |
| `HTTP 404` | `{"detail":"File not found"}` | 28 | sediment (`file_…`) の実体消失 |
| `signed_HTTP 404` | (空) | 1 | 署名 URL 取得後の CDN 失敗 |
| `HTML_response` | (HTML body 先頭) | 1 | backend がエラー HTML を返した稀ケース |

判明事項:

1. **`no_signed_url` は 99% が `error_code=file_not_found`** → user upload は概ね 6〜12 ヶ月で実体削除される。single で再取得しても復旧不能。
2. **`permission_error` は 1 件のみ** (composite asset / 2023-11 の古い会話)。共有取り消しの疑い。
3. **ID prefix と reason の対応は完全**: `file_…` (sediment) → `HTTP 404`、`file-…` (user upload) → `no_signed_url`。
4. **会話作成月と失敗率の相関**: 3 ヶ月以内は失敗ほぼゼロ、6〜12 ヶ月で `no_signed_url` が常態化、12 ヶ月超でほぼ確実に削除。

この結果を受けて [README.md](../README.md) §トラブルシューティング の reason 説明を実機ログに合わせて修正した (旧版は HTTP 404 と no_signed_url の対応が逆だった)。

## 5. `_bulk-asset-failed 2.log` 分裂バグの発見

3 ラン完了後に `_bulk-asset-failed.log` を確認すると、Run C 分の 19 件しか残っておらず Run A/B の 128 件は別ファイル `_bulk-asset-failed 2.log` に切り離されていた:

```
-rw-------@ 26463 May 12 23:11 _bulk-asset-failed 2.log    ← 128 件 (Run A+B)
-rw-------@  4221 May 12 23:22 _bulk-asset-failed.log      ← 19 件 (Run C)
```

### 原因推定

`appendAssetFailedLog` ([chat-bulk-export.js:798](../chat-bulk-export.js#L798)) は read-then-rewrite パターンで実装されており、ロジック上は append 動作するはず。ただし以下の要因で session 跨ぎでファイル分離が起こる:

- Console 貼り付け実行 (= browser session) ごとに `rootDir` handle が新規生成される
- 同じフォルダを picker で選び直しても、内部 handle は別オブジェクト
- File System Access API の `createWritable()` は内部的に temp file (`.crswap`) で書き、close 時にリネーム
- macOS Documents/ は iCloud Drive 同期対象で、複数 session からの rapid rewrite が rename レースを起こした可能性

### 影響範囲

manifest / queue / run-stats では同じ現象は発生していない (session ごとに 1 回程度しか書かれない)。`_bulk-asset-failed.log` だけが**短時間内に頻繁に rewrite される**ため symptom が出やすかった。

詳細は [docs/observation-experiment-1.md §補足: v0.8.17 で発見した log 書き込みのバグ](observation-experiment-1.md) を参照。

## 6. v0.8.18 — mergeOrphanLogDuplicates の実装

### 設計

起動時に `rootDir.entries()` を走査し、`<stem> N.<ext>` パターンに一致する duplicate ファイルを検出 → main file へ append → duplicate 側は空に truncate する。

```js
const mergeOrphanLogDuplicates = async (baseName) => {
  // baseName = "_bulk-asset-failed.log" → match "_bulk-asset-failed 2.log",
  // "_bulk-asset-failed 3.log" など。
  ...
};
```

### 対象

- `_bulk-asset-failed.log`
- `_bulk-failed.log`

起動時の 1 回のみ実行。duplicate が無ければ silent skip (log 出力なし)。

### 制限

- duplicate ファイルを削除する権限が無いケースに備えて空 truncate で済ませる
- run 中に rename レースで新たな duplicate が生まれることまでは防げない (次回起動時にマージされる)
- 根本原因 (FSA rename レース) は browser 側 / macOS 側の問題で、ユーザー側コードからは回避不能

### 動作確認

v0.8.18 を v0.8.17 完走後の状態 (duplicate 無し) で実行:

```
Run 5 (v0.8.18, 2026-05-12 16:04:46Z): elapsed=0 / skipped=636 / mergeOrphanLogDuplicates silent skip
```

[_bulk-run-stats.jsonl](../) 5 行目で確認。
