# 観察実験 1: 休止時間と burst capacity の関係

[docs/throttle-burst-investigation.md](throttle-burst-investigation.md) §4.6 で計画した観察実験を、2026-05-10 〜 2026-05-11 の bulk export 2 ディレクトリから既存 manifest を集計して実施。新しい run は不要。

## 入力

- `chatgpt-export-v0.8.13/_bulk-manifest.json` (543 件、status=done 543) — 最終更新 2026-05-11 05:21 JST
- `chatgpt-export-v0.8.14/_bulk-manifest.json` (399 件、status=done 399) — 最終更新 2026-05-11 12:00 JST

両 manifest とも全エントリが status=done で failed が無い。各エントリの `exportedAt` (ISO 時刻) を時系列ソート。

## 手法

連続する 2 件の export 完了時刻の差を `gap` とし、**`gap ≥ 10 分`** を run 境界と判定する。

10 分閾値の根拠: conversation cooldown ladder は v0.8.15 で 60/90/120/180/240/300 秒 (最大 5 分)。1 回の cooldown では 5 分を超えないが、連続 cooldown と delay の合算を考慮して 10 分マージンを確保。それ未満の gap は throttle 起因の intra-run cooldown とみなして同一 run に含める。

各 run について `(直前の run 終了 → 今回の run 開始) の gap` と `今回の run の成功件数 = capacity` を抽出。

## 回帰

### 線形モデル

```
capacity ≈ 1.23 × gap_min + 16.5
R² = 0.989    N = 7
```

予測値:

|---:|---:|

|  60 |  90.4 |

| 120 | 164.2 |
| 150 | 201.1 |
| 180 | 238.0 |
| 240 | 311.8 |

### 飽和モデル

```
capacity ≈ 400 × (1 - exp(-gap / 200))
R² = 0.987    N = 7
```

| gap (分) | 飽和予測 (件) |
|---:|---:|

両モデルは観測範囲 (15〜189 分) で**ほぼ同じ予測**を出すが、240 分以上で差が出始める (線形 311 件 vs 飽和 279 件)。

## 結論

### H1 (休止時間 → capacity) は強く支持される

- R² = 0.989 (線形) / 0.987 (飽和) は N=7 にしてはかなり強い
- 観測範囲では「休止 1 分あたり約 1.2 件の capacity」がほぼ線形に増える
- 初回 run の小ささ (26 / 35 件) は「事前休止が観測できないが、何時間も空いていたわけではない」状態と整合し、回帰モデルの低 gap 領域 (~30 件の切片) と一致

### 線形 vs 飽和の判別はまだできない

- 観測範囲は 15〜189 分。190 分以上のデータが存在しない
- 線形なら 240 分で約 312 件、飽和なら 280 件と予測がズレる
- 「3 時間以上空けても更に伸びるのか、それとも 250 件付近で頭打ちなのか」が次の問い

### H2 (時刻軸) について

データ点の run 開始時刻:

- 03:10 JST → capacity 238 (gap 182 分)
- 04:39 JST → capacity 44 (gap 32 分)
- 05:10 JST → capacity 30 (gap 20 分)
- 05:32 JST → capacity 35 (gap N/A)
- 06:10 JST → capacity 40 (gap 15 分)
- 07:06 JST → capacity 73 (gap 31 分)
- 10:57 JST → capacity 251 (gap 189 分)
- 23:16 JST → capacity 205 (gap 152 分)

直前 gap に強く支配されており、**時刻単独で capacity が変わる強い兆候は見えない**。深夜帯 (23:16, 03:10) のスコアが高いのは事実だが、これらは長い gap (152, 182 分) を伴う。10:57 JST (gap 189 分) も同等の capacity を出している。**H2 は本観察データの範囲では支持されない**。

ただし「日中に 3 時間休止」のサンプルが 1 点 (10:57) しかないので、H2 の完全棄却には日中サンプルを増やす必要がある。

### H3 (asset/サイズ非依存) について

run ごとの asset 数や会話サイズの相関は本観察では分析せず。manifest には `stats.saved` (保存 asset 数) と `stats.jsonAssetCount` が含まれているため、別途分析可能。今回はスコープ外。

## 次の能動実験への含意

### 推奨: 実験 1 の改訂版

オリジナル計画 (60 / 120 / 180 分の 3 ポイント) は線形仮説の確認には充分だが、**線形 vs 飽和の判別には不十分**。次の 4 ポイントを推奨:

| `t_rest` (分) | 目的 | 線形予測 | 飽和予測 |
|---:|---|---:|---:|
|  60 | 中点確認 |  90 | 104 |
| 120 | 中点確認 | 164 | 181 |
| 180 | 既知点との照合 | 238 | 237 |
| **300** | **線形 vs 飽和の判別** | **385** | **311** |

300 分点で 70 件以上の差が出るので、1 ポイントで決着できる。

### コスト見積もり

- 各 run は capacity 上限 + 内部 cooldown でおよそ 60-90 分かかる
- 計画休止 4 ポイント (60 + 120 + 180 + 300 = 660 分 ≈ 11 時間) + run 実行 4 × 75 分 = 5 時間
- **合計 16 時間程度**。1 日では収まらないので 2 日に分けるのが現実的

### 代替: 「とりあえず 300 分点だけ」の最小版

最小コストで仮説の中で最も価値ある一点 (= 300 分以上) を取る:

1. quota を完全リフィル (12 時間以上の休止)
2. 1 run 走らせて capacity 計測
3. 結果が >310 件 → 線形寄り。<290 件 → 飽和寄り。中間ならグレー

これだけなら **半日で 1 点**追加でき、N=8 になる。最初の能動実験としてはこれが効率的かもしれない。
